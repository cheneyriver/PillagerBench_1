# PillagerBench 项目说明与实验复现指南（中文版）

本文面向希望**从零跑通实验**的读者，用通俗语言说明「这个项目在做什么」「每个文件夹大概干什么」，并给出**可直接复制执行**的命令步骤。

---

## 一、这个项目是做什么的？

**一句话：** 在《我的世界》（Minecraft）里，让多个由**大语言模型（LLM）控制的智能体**分成两队比赛，用一套标准流程测量它们的表现，这就是 **PillagerBench**。

**再展开一点（高中生也能懂）：**

- **为什么要用 Minecraft？**  
  游戏里要采集、合成、移动、和对手互动，状态很复杂，很像「小型真实世界」，适合测试 AI 会不会**协作、规划、对抗**。

- **「Benchmark（基准测试）」是什么意思？**  
  就像考试用同一张卷子：大家用**相同的地图规则、相同的对手类型、相同的计分方式**，这样不同模型、不同算法的结果才可以**公平比较**。

- **论文与引用：**  
  项目对应论文 *PillagerBench: Benchmarking LLM-Based Agents in Competitive Minecraft Team Environments*（IEEE CoG 2025），预印本：<https://arxiv.org/abs/2509.06235>。

## 二、你的代码目录里，主要文件夹各自干什么？

下面假设你正在 **`PillagerBench_1`** 这一层（里面有 `main.py`、`configs/`、`bench/` 等）。若你解压路径不同，请**先进入包含 `main.py` 的那一层**再执行命令。

| 路径（相对 `PillagerBench_1`） | 通俗说明 |
|------------------------------|----------|
| `main.py` | **程序入口**：读取 Hydra 配置，启动整套基准测试。 |
| `api_keys.py`（需自建） | **密钥文件**：存放 OpenAI、DeepSeek、OpenRouter 等 API 密钥；**不要上传到公开仓库**。 |
| `configs/` | **实验开关**：用 YAML 写「跑哪些场景、用哪个智能体、跑几局、模型名是什么」。`benchmark.yaml` 是默认完整基准配置；`test_*.yaml` 多为论文里不同模型/消融实验；`agents/` 里是各智能体（如 Tacticrafter）的模型与温度等参数。 |
| `bench/` | **基准测试核心逻辑**：启动本地 Minecraft 服务端、创建环境、调度红队/蓝队智能体、记录日志等。 |
| `bench/mc_server/` | **本地 Minecraft 服务器**：需要 **`server.jar`**（官方服务端，见下文下载说明）、`eula.txt`、世界存档文件夹（如 `Mushroom War`、`Dash and Dine`）等。 |
| `agents/` | **智能体实现**：例如 `tacticrafter.py`（主方法智能体）、各类对手 `mushroom_war_*`、`dash_and_dine_*`、`do_nothing`（几乎不动的基线）等。 |
| `scenarios/` | **具体游戏规则**：例如「蘑菇战争」「冲刺就餐」的场景描述、出生点、胜负条件等（Python + 部分 JS）。 |
| `voyager/` | **与游戏交互的底层**：Mineflayer（Node.js）桥接、动作原语、提示词（prompts）、LLM 调用等，多来自/改编自 Voyager 类框架。 |
| `voyager/env/mineflayer/` | **Node 侧依赖**：需执行 `js_setup.sh` 安装 npm 包并编译子项目。 |
| `requirements.txt` | **Python 依赖列表**（需配合 Python 3.10 等）。 |
| `js_setup.sh` | **一键安装前端/机器人侧依赖**（在 Linux/macOS 下使用）。 |
| `compose.yaml` + `Dockerfile` | **可选**：用 Docker 搭环境（镜像名 `pillagerbench`，映射端口 `49172`）。 |
| `collate_results.py` | **结果汇总与画图**：读取某次运行产生的 `logs` 目录，做统计与可视化（需在脚本里**改路径**指向你的日志）。 |

运行后，Hydra 会把本次任务的输出目录设为**带时间戳的路径**（见 `configs/benchmark.yaml` 里的 `hydra.run.dir`），其中会包含各场景的 `logs/` 等。

## 三、复现前你需要准备什么？

### 3.1 硬件与系统

- 能运行 **Java 17**、**Node.js 20**、**Python 3.10** 的机器（Linux / macOS / Windows 均可；下文以 **Linux 本地安装**为主）。
- 跑完整 `benchmark` **耗时很长**（大量对局 × 多次调用大模型），且会产生 **API 费用**；建议先用少量 `num_episodes` 或单独一个小配置试跑。

### 3.2 必须准备的文件与账号

1. **API 密钥**（至少一种，取决于你要用的模型）：  
   - OpenAI（如 GPT-4o）  
   - DeepSeek  
   - OpenRouter（可接入多种模型）  
   可选：本地 **Ollama**（见 `configs/agents/tacticrafter_local_*.yaml` 一类配置）。

2. **`bench/mc_server/server.jar`**  
   本仓库目录里**未必包含**该文件；你需要自行放入与项目兼容的 **Minecraft 服务端**（README 写明客户端连接版本为 **1.19.4**，服务端应使用**匹配的 1.19.4 server jar**）。  
   - 从 Minecraft 官方下载对应版本的 `server.jar`，放到：  
     `PillagerBench_1/bench/mc_server/server.jar`  
   - 首次运行前通常需要已接受 EULA；本仓库里已有 `eula.txt`，若服务端仍提示，把其中 `eula=true` 即可。

3. **（可选）用正版客户端旁观**  
   README 说明可用 **Minecraft 1.19.4** 连接 **`localhost:49172`**（端口可在 `configs` 里改 `mc_port`）。

## 四、本地安装：逐步命令（推荐按顺序执行）

以下命令默认你在 **`PillagerBench_1`** 目录下（即 `main.py` 所在目录）。

### 步骤 1：进入项目目录

```bash
cd PillagerBench_1
```

把 `/path/to/` 换成你机器上的真实路径。

### 步骤 2：创建 `api_keys.py`

在与 `main.py` **同一目录**新建文件 `api_keys.py`，内容模板如下（把引号里换成你的真实密钥；**不用的可以写成空字符串 `""`，但变量名必须保留**）：

```python
openai_api_key = "你的OpenAI密钥"
deepseek_api_key = "你的DeepSeek密钥"
openrouter_api_key = "你的OpenRouter密钥"
```

`main.py` 会把它们写入环境变量 `OPENAI_API_KEY`、`DEEPSEEK_API_KEY`、`OPENROUTER_API_KEY`。

### 步骤 3：放置 `server.jar`

确认文件存在：

```bash
ls -la bench/mc_server/server.jar
```

若不存在，请按上一节说明下载 **1.19.4** 的 `server.jar` 放到该路径。

### 步骤 4：安装 Node 依赖（Mineflayer）

```bash
chmod +x js_setup.sh
./js_setup.sh
```

该脚本会进入 `voyager/env/mineflayer` 执行 `npm install`，在子目录里 `npx tsc` 编译，再安装外层依赖。

### 步骤 5：创建 Python 虚拟环境并安装依赖

```bash
python3.10 -m venv venv
source venv_v1/bin/activate
pip install -U pip
pip install -r requirements.txt
```

若系统默认已是 Python 3.10，可把 `python3.10` 改成 `python3`。

### 步骤 6：运行基准测试

默认配置名为 **`benchmark`**（对应 `configs/benchmark.yaml`）：

```bash
source venv_v1/bin/activate
python main.py -cn benchmark
```

**说明：**

- `-cn benchmark` 表示使用 `configs/benchmark.yaml`。  
- 你也可以换成 `test_gpt-4o`、`test_o3-mini`、`test_self_play` 等，只要 `configs/` 里有同名 yaml：  
  `python main.py -cn test_gpt-4o`  
- 某类实验会引用 `configs/agents/` 下的不同智能体配置（例如 `tacticrafter_o3-mini.yaml`），由对应 `test_*.yaml` 的 `defaults` 决定。

运行完成后，在 Hydra 生成的时间戳目录下可找到日志（具体路径以终端输出为准，形如 `./logs/年-月-日/时-分-秒/`）。

## 五、用 Docker 复现（可选）

若你更熟悉 Docker，可按官方 README 思路操作（**注意**：Windows 下脚本名是 `js_setup_docker.ps1`，Linux 下通常仍可在容器内执行 `bash js_setup.sh`）。

1. 安装 Docker。  
2. 在 **`PillagerBench_1`** 目录放好 `api_keys.py` 与 `bench/mc_server/server.jar`。  
3. 构建镜像（README 示例）：  
   `docker build -t PillagerBench .`  
4. 安装 Node 依赖（README 用 PowerShell；Linux 可用 compose 运行一次并执行 `js_setup.sh`）：  
   参考 `js_setup_docker.ps1` 等价于：`docker compose run --rm pillagerbench bash js_setup.sh`  
5. 启动并进入容器：  
   `docker compose up -d`  
   `docker attach pillagerbench`  
6. 在容器内（工作目录一般为 `/PILLAGER/`）：  
   `python main.py -cn benchmark`

`compose.yaml` 将本目录挂载到容器内 `/PILLAGER/`，并映射端口 **49172**。

## 六、结果整理与画图

1. 找到某次运行生成的 **`logs` 目录**（在带时间戳的输出文件夹里）。  
2. 打开项目根目录下的 **`collate_results.py`**。  
3. 在文件末尾的 `main()` 里，修改 **`logs_dirs`** 列表：把里面的路径改成**你本机**上真实的 `logs` 路径（可参考脚本里注释掉的例子）。  
4. 在已激活的 venv 中执行：

```bash
python collate_results.py
```

脚本会读取各场景下的 `scenario.yaml`、奖励文件等，并可选绘制热力图、奖励曲线等（需安装 `matplotlib`、`pandas` 等，已在 `requirements.txt` 中）。

### 6.1 复现运行后，结果具体放在哪里？

一次运行通常会在 **Hydra** 设定的工作目录下生成（本项目在 `configs/benchmark.yaml` 里配置为 `hydra.run.dir: ./logs/${now:%Y-%m-%d}/${now:%H-%M-%S}`）。

你可以按下面“相对目录形状”来找结果（以你的实际运行输出目录为准）：

```text
logs/scenario{scenario_i}/
  scenario.yaml  # 本次“场景-配置”的快照（场景名、红队智能体名、对手名等）
  episode{episode_i}/
    rewards/red_team.txt
    rewards/blue_team.txt
    (可选) red_team/code.json  # 如果存在，用于统计 rollout_num_iter 等额外信息
```

如果你只想看“最终得分”，通常就是：

- `.../episode{episode_i}/rewards/red_team.txt` 的第 121 行（第 120 秒，索引 120）
- `.../episode{episode_i}/rewards/blue_team.txt` 的第 121 行（同理）

### 6.2 `benchmark/test_*.yaml` 里的关键参数怎么理解？

你执行 `python main.py -cn benchmark` 时，Hydra 会读取 `configs/benchmark.yaml`，并在内部把“智能体配置”（`configs/agents/*.yaml`）组合进来。

`configs/benchmark.yaml` 里的常见字段：

- `mc_port`：Minecraft 连接端口（默认 `49172`）。
- `server_port`：程序启动多个机器人环境用的起始端口（会不断递增）。
- `env_wait_ticks`：环境复位后的等待 tick 数，让状态稳定。
- `scenarios:`：要跑哪些“比赛场景”，每个条目是一组对局集合。

`scenarios` 每个条目里的关键字段：

- `name`：场景名（例如 `Mushroom War`、`Dash and Dine`），决定游戏规则。
- `agent`：红队智能体名（例如 `tacticrafter` / `cot_agent` / `random`）。
- `opponents`：对手列表。因为默认是两队对抗，这里通常只用到 `opponents` 列表的第一个（蓝队）。
- `num_episodes`：跑几局（代码里会循环 `range(num_episodes)`）。
- `kwargs:`：场景构造函数参数；当前最重要的是 `episode_timeout`：一局中（当 episode 已开始后）智能体执行代码的最长时间；超时会导致 bot 说 `[episode timeout]`，从而影响得分。

在 `configs/test_*.yaml` 里，你还会看到：

- `defaults: ... agents: [某个agents yaml名]`：这表示同样的 `scenarios`，但红队智能体使用不同模型/参数（比如 `tacticrafter_o3-mini` vs `tacticrafter`）。

`configs/agents/tacticrafter*.yaml` 的典型字段（以 `tacticrafter.yaml` 为例）：

- `negotiator_model_name` / `causal_model_name` / `opponent_model_name`：分别控制“谈判/因果/对手建模”用的 LLM 模型。
- `*_temperature`：温度，控制随机程度（越大越“发散”）。
- `critic_mode`：批判/评价模式（这里是 `auto`）。
- `voyager_options:`：Mineflayer 动作执行相关参数（例如 `action_agent_temperature`、`action_agent_task_max_retries`、是否显示聊天日志等）。

### 6.3 `collate_results.py` 怎么读取结果并计算分数？

`collate_results.py` 对每个你在 `main()` 里写进 `logs_dirs` 的目录做如下事情：

1. 读取 `logs_dir/../main.log`（可选，用于统计 AI 调用耗时、Token 等；如果没有，脚本仍会继续读取奖励文件）。
2. 遍历 `logs_dir` 下每个 `scenario_name/` 目录，找到其中的 `scenario.yaml`。
3. 再遍历每个 `episode{n}/` 目录，读取：
   `episode_path/rewards/red_team.txt` 和 `episode_path/rewards/blue_team.txt`。
4. 认为这两个文件每行是一段时间（按秒）对应的积分序列：脚本取前 `121` 行（0 到 120 秒，共 121 个点），最终比较使用 `index 120`（第 120 秒积分）。

所以你在图/表里看到的“胜负”和“分差”，就是：

- `diff = red_points[120] - blue_points[120]`
- `winrate`：`diff>0` 红队赢记 1；`diff<0` 记 0；相等记 0.5。

此外，脚本里还有一个重要细节：它会用 `points_scale_dict` 对不同场景做缩放（默认 `Mushroom War: 1`，`Dash and Dine: 0.1`），这样不同场景画在一起时数值量纲更可比。

### 6.4 如何解读不同的结果图？

`collate_results.py` 默认会画三类关键图，并在控制台输出一些数值摘要：

1. **热力图三联（通常是 Points / Sabotage / Win rate）**
   横轴是蓝队对手名字，纵轴是红队智能体名字；Points（红队积分）格子值越大表示红队平均积分越高；Sabotage（破坏力）表示相对“蓝队正常得分基线”，蓝队少拿了多少分，值越大通常压制越强；Win rate（胜率）越接近 1 红队越强，越接近 0 红队越弱。

2. **Points Over Time（随时间的积分曲线）**
   横轴是秒数，纵轴是红队积分；线越高代表在那个时间点更领先，曲线形状能看出“早期领先还是后期反超”。

3. **Points Difference over Episode（每局的积分差）**
   横轴是 episode 编号，纵轴是 `red - blue` 的积分差（同样取第 120 秒）；越接近或大于 0 表示红队越占优势，波动越大说明表现越不稳定。

4. **控制台打印的额外统计（用来判断稳定性与成本）**
   `Average AI invoke time / input tokens / output tokens`：LLM 调用的耗时与 Token，影响运行成本和速度；`Average rollout_num_iters`：底层规划迭代相关指标，通常越大表示策略生成过程更复杂/更耗时；`Average dedupe ratio`：事件去重相关指标，帮助你判断是否在重复处理类似状态。

如果你把这些图和 `episode_timeout` / 模型配置（`tacticrafter*.yaml`）对应起来，就能快速看出：改了什么配置后，红队是更强、更稳定，还是只是更耗时。

## 七、如何「完整复现论文式实验」？

论文中的完整对比涉及**多种模型与多种基线/消融**，对应仓库里多个 Hydra 配置，思路是：

1. **同一套场景与局数**：以 `configs/benchmark.yaml` 为模板，或直接使用各 `test_*.yaml`。  
2. **分别运行不同配置**，每次得到独立时间戳目录，避免互相覆盖。  
3. **用 `collate_results.py` 汇总**多次运行的 `logs` 路径。  

具体要跑哪些 yaml，请以你关心的论文图表为准；仓库中常见文件名含义如下（**仅供参考**）：

- `test_gpt-4o.yaml`、`test_o3-mini.yaml`、`test_gemini-2.yaml`：不同云端模型。  
- `test_mushroom_war_baselines.yaml`、`test_dash_and_dine_baselines.yaml`：场景相关的基线实验。  
- `ablation_no_causal.yaml`、`ablation_no_opp.yaml`：消融（去掉因果模块 / 对手建模等）。  
- `test_self_play.yaml`、`test_self_play_improvement.yaml`：自对弈相关。  
- `test_random.yaml`、`test_cot.yaml`：随机策略、思维链等对照。

智能体侧细节在 `configs/agents/tacticrafter*.yaml`（模型名、温度等）。

## 八、常见问题排查

| 现象 | 可能原因 | 建议 |
|------|----------|------|
| 提示找不到 `api_keys` | 未在 `main.py` 同目录创建 `api_keys.py` | 按步骤 2 创建并确认变量名一致 |
| Minecraft 服务端起不来 | 缺少 `server.jar` 或 Java 版本不对 | 放入 1.19.4 `server.jar`，安装 Java 17 |
| Node/Mineflayer 报错 | 未执行或未成功执行 `js_setup.sh` | 重新运行；确认 Node 为 20 左右 |
| 连接游戏失败 | 端口被占用或防火墙 | 检查 `mc_port`（默认 49172）是否空闲 |
| API 报错 / 额度不足 | 密钥错误或模型名与账号权限不匹配 | 检查 `api_keys.py` 与 `configs/agents/*.yaml` 中的模型名 |
